Building a Real-Time Japanese Learning Companion for Zelda

A developer's journey through camera feeds, OCR, local LLMs, and NLP

What was built

A Python script running on a MacBook Pro M1 Pro that reads a live camera feed from a Samsung S21 pointed at a TV, extracts Japanese dialogue from Breath of the Wild using Apple's native OCR framework, passes it through local NLP libraries for linguistic analysis, sends only the translation task to a local LLM via Ollama, and serves a real-time web UI at localhost:5002. Two modes: TRANSLATE for fast live translation while playing, and LEARN for a full word-by-word lesson with romaji, grammar notes, kanji breakdowns, vocab tracking with familiarity tiers, and a spaced-repetition quiz system. Entirely local. Entirely free. Built iteratively over multiple weeks.

1. Introduction — The Problem and the Motivation

The goal was simple on paper: play The Legend of Zelda: Breath of the Wild entirely in Japanese and use it as a living Japanese textbook. Every time an NPC speaks, instead of skipping past the dialogue, understand it. Every kanji encountered repeatedly should become familiar over time. By the end of the game, you should have meaningfully progressed in the language — passively, through play.

No off-the-shelf tool handled this specific scenario. Browser extensions, phone apps, and OCR tools all assumed you were reading text on a screen you controlled. The game was on a TV. The dialogue was rendered in Zelda's own stylised font. The text animated letter by letter. The entire pipeline had to run locally — no cloud APIs, no subscription fees, no sending game footage to a third-party server.

This post traces the full development arc: from the first architecture through each major pivot, then through the detailed engineering of every component from camera input to web interface.

2. Architecture Evolution — The Three Major Pipelines

The system went through three substantially different architectures before arriving at the final design. Understanding these iterations explains every technical choice in the current codebase.

2.1 Architecture 1 — OCR + Text LLM (Tesseract + qwen2.5:7b)

S21 camera → OpenCV crop → Tesseract OCR → qwen2.5:7b → Flask UI

The first architecture was the most natural starting point: a dedicated OCR engine extracts Japanese text from the camera frame, then a text LLM translates and explains it.

LayerTool
CameraSamsung S21 via IP Webcam (MJPEG over WiFi)
Frame grabOpenCV
OCRTesseract (Japanese language model)
Translationqwen2.5:7b via Ollama
UIFlask + browser polling

The problem was Tesseract. It produced garbage output on every frame — not occasionally, consistently. The game's stylised font on a semi-transparent dialogue box sitting on top of active gameplay was beyond what Tesseract could handle. Extensive preprocessing (HSV masking, grayscale thresholding, dilation, character count filters, Japanese character validation) helped at the margins but never produced stable output. Tesseract is designed for printed document fonts.

2.2 Architecture 2 — Pure Vision LLM (qwen2.5vl:7b — tried and abandoned)

S21 camera → OpenCV crop → qwen2.5vl:7b (image → translation) → Flask UI

With Tesseract failing, the pivot was to eliminate OCR entirely. Vision-language models receive an image directly and return both extracted text and translation in a single call. qwen2.5vl:7b handled the stylised Zelda font far better than Tesseract, understood context, and dealt with furigana naturally.

The hardware reality on a 16 GB M1 Pro killed the approach:

ComponentRAM
Text transformer (Q4_K_M)~4.5 GB
Vision encoder (F16, unquantized)~9.0 GB
KV cache~0.5 GB
Total~14 GB

The vision encoder stores all 32 of its transformer layers in full F16 precision — a property of the model architecture, not user-configurable. 14 GB on a 16 GB machine left almost nothing for the OS. ollama ps showed a 62%/38% CPU/GPU split. On Apple Silicon, any significant CPU fraction means weights are being paged off unified memory. Normal inference: 9 seconds. Under swap pressure: 60 seconds. At worst: memory_pressure showed 54 MB free RAM with 1.7 million swap-outs.

Every tuning parameter was tried: num_ctx reduced to 1024, num_batch set to 1, OLLAMA_FLASH_ATTENTION enabled, OLLAMA_GPU_LAYERS tried (has no effect on Apple Silicon — this flag is for NVIDIA/AMD discrete GPUs only). None solved the fundamental constraint: a 14 GB model on a 16 GB machine will always split CPU/GPU.

LLaVA-Phi3 was tested as a lighter alternative (~2.9 GB). Japanese OCR quality was too poor to be useful. The pure vision LLM architecture was abandoned.

2.3 Architecture 3 — OCR + Text LLM + NLP Libraries (Final)

Apple Vision + qwen2.5:7b + fugashi/pykakasi/jamdict

S21 camera → OpenCV crop → preprocess → Apple Vision OCR → fugashi + pykakasi + jamdict + rule-based grammar → qwen2.5:7b (translation only) → Flask UI (TRANSLATE + LEARN modes)

The final architecture gives each task to the tool best suited for it:

ToolTaskLatency
Apple VisionImage → Japanese text~100 ms
fugashi/MeCabText → tokens + POS tags~50 ms
pykakasiTokens → romaji~10 ms
jamdictTokens → meanings~100 ms
Rule-basedPOS → grammar patterns~5 ms
qwen2.5:7bJapanese → English (translation only)~2,500 ms

The LLM now does exactly one thing: translate. Every other part of the lesson is generated by deterministic libraries that are faster and more accurate than an LLM for those specific tasks.

3. Component — Camera Input

3.1 Capture Card vs. Phone Camera

A capture card would have been the cleanest technical solution — stable 1080p digital signal from the Switch dock, fixed pixel coordinates, no lighting variation or compression. But it costs money, requires USB to the laptop, and adds hardware to manage. The phone approach is free, wireless, and uses existing hardware. The trade-offs are real but acceptable: camera angle can drift, glass introduces glare, JPEG compression adds per-frame noise. The code accepts any MJPEG video source — swapping to a capture card means changing one URL constant.

3.2 scrcpy — Tried and Abandoned

The first attempt was scrcpy, a tool that mirrors an Android device's screen over ADB. Full setup was completed: Developer Options enabled on the S21, wireless ADB pairing, scrcpy showing the camera feed in a window. The problem came when trying to read from Python.

scrcpy registers its output as a virtual camera device on macOS, but OpenCV read only black frames. CAP_AVFOUNDATION was tried explicitly. Device indices 0 and 1 were tested with color checks. Camera permissions were verified in System Settings. The device reported ret=True and a valid 1080×1920 resolution, but every frame had mean color [0, 0, 0] — pure black.

The --no-window flag was tried to run scrcpy headlessly. This immediately disabled the video stream: "No video playback, no video mirroring." scrcpy's virtual camera only works while it is actively displaying in a window. OBS Virtual Camera was identified as a proper workaround, but that added another dependency. The entire scrcpy path was abandoned.

3.3 IP Webcam — The Working Solution

IP Webcam turns the S21 into an MJPEG HTTP server on the local network. OpenCV reads it directly via URL — no ADB, no virtual camera, no window focus issues:

cap = cv2.VideoCapture("http://192.168.1.107:8080/video")

The connection was validated with a minimal test script before wiring into the main pipeline. DroidCam was evaluated but its free tier caps at 480p — complex kanji glyphs at 480p lose strokes and become ambiguous to OCR. IP Webcam streams at full phone resolution.

3.4 Capture Loop Design

The capture loop runs in a dedicated daemon thread. A key early discovery: macOS Cocoa constraints mean cv2.imshow() cannot be called from a background thread — attempting this throws immediately. Rather than restructuring threading, the debug preview was changed to writing ~/Downloads/preprocessed_crop.jpg on every iteration. macOS Preview auto-refreshes open files, giving a live view of exactly what the OCR engine receives with no threading complexity. This pattern was used throughout development whenever OCR behaviour needed to be diagnosed.

4. Component — Preprocessing

4.1 The Input Challenge

Zelda's dialogue box is white text on a dark semi-transparent rounded rectangle sitting on top of active gameplay. The camera adds angle distortion, JPEG compression, and occasional glare. Preprocessing needed to isolate clean text pixels without destroying the characters in the process.

4.2 Finding the Dialogue Box

The first crop attempt used hard-coded percentages. It worked but broke whenever the camera moved. An LLM "discovery mode" was tried — sending the full frame to the vision model every 2 seconds to get dialogue box coordinates as JSON. Detection calls took ~55 seconds and returned inconsistent bounds. Unusable.

The permanent solution was calibrate.py: a one-time setup tool that grabs a frame from the camera, opens it in an OpenCV window using cv2.selectROI, lets the user draw the bounding box, and saves it to bounds.json. The main script reads this on startup and fails immediately if it's missing.

4.3 Image Preprocessing Pipeline

Attempt 1 — HSV Masking ✗

Isolate white pixels using HSV colour space. The dark dialogue background fell in the same HSV range. Entire output was white. Abandoned.

Attempt 2 — Grayscale Threshold at 180 ✗

The dialogue background sits at gray values 40–80; text is at 220+. Thresholding at 180 cuts between them. Added 2×2 dilation to reconnect strokes broken by JPEG compression. Better, but also caught bright game-world pixels (sunlight, fire).

Attempt 3 — CLAHE ✗

Contrast Limited Adaptive Histogram Equalisation sharpened text strokes but amplified background texture, producing more OCR noise. Abandoned.

Attempt 4 — White-on-Black Extraction ✓ (Final)

Rather than trying to isolate the dialogue box while preserving game context, reduce the entire image to its essential signal: everything above brightness 200 becomes white; everything else becomes black. Combined with edge kill (zero out 8 px left/right margins), 2× INTER_CUBIC upscale, a 20 px solid black padding border, and JPEG quality set to 100 to preserve dakuten dots. The threshold was later tuned from 200 to 160 for scenes where Zelda uses a grunge/eroded font style.

4.4 Furigana Masking

Zelda uses furigana — small kana printed above kanji to show their reading. Every OCR engine read these as separate lines of text, producing garbled output. Several approaches were tried:

Attempt 1 — Prompt Engineering ✗

Tell the model to ignore small characters. Failed — 3b models couldn't follow this reliably across varied frames.

Attempt 2 — Blob Size Filtering ✗

Find all text contours, filter out small ones. Background noise dominated the contour list, making median character height unreliable.

Attempt 3 — Row Density (Initial) ✗

Sum bright pixels per horizontal row to build a density profile, find peaks, black out furigana gaps. Failed on short single-line dialogue where furigana rows crossed the max-based threshold.

Attempt 4 — Walk-Back Algorithm ✗

Detect where each text band starts, walk upward to find furigana, black out only that region. Produced too many edge cases for reliable production use.

Attempt 5 — Row Density with Median ✓ (Final)

Same peak detection but replacing max with median for the threshold:

non_zero = row_density[row_density > 0]
median_density = float(np.median(non_zero))
furigana_threshold = median_density * 0.42

Using max was skewed by one very dense row. median is stable regardless of sentence length or line count. The 0.42 multiplier was tuned by trial on real game frames.

💡 Biggest win
Moving the phone camera closer to the TV was more effective than any amount of image preprocessing at eliminating OCR errors. Better source material beats better compensation downstream.

5. Component — OCR Engine

5.1 Tesseract

The starting OCR engine. Trained on printed document fonts. Produced garbage output on Zelda's stylised dialogue font. The jpn_vert (vertical Japanese) model was also tried — no meaningful improvement. Tesseract was not designed for game UI.

5.2 manga-ocr

manga-ocr is a ~400 MB specialised Japanese OCR model fine-tuned on manga images. Game fonts are stylistically closer to manga than printed documents. Two preprocessing scenarios were tested. In both cases hallucination was a significant problem — with the near-raw crop it produced complete hallucinations (tokens like 定款, チェックス, 鹿児クエ with none of the actual dialogue); with the full white-on-black preprocessed image, it was trained on black ink on white paper, so the inverted distribution caused continued hallucination. manga-ocr also added ~638 ms per call versus Apple Vision's ~100 ms. Dropped.

5.3 Apple Vision Framework (Final)

Apple Vision (VNRecognizeTextRequest) is built into macOS and runs on the M1 Neural Engine — zero RAM footprint, zero model download, Japanese supported natively. Integration via pyobjc:

import Vision
request = Vision.VNRecognizeTextRequest.alloc().init()
request.setRecognitionLanguages_(["ja"])
request.setUsesLanguageCorrection_(False)

setUsesLanguageCorrection_(False) is critical — with autocorrect enabled, macOS would mangle archaic NPC speech (かのう, のう, じゃ) trying to normalise it to modern Japanese. OCR timing: ~100 ms on M1 Neural Engine.

Why large models read game images easily
Large multimodal models process images contextually — they understand what a dialogue box is, can distinguish furigana from main text by knowing what furigana is, and use surrounding context to disambiguate ambiguous characters. Local OCR engines are pixel pattern matchers with no semantic understanding of what they are looking at.

5.4 OCR Stability — Consensus Mechanism

Apple Vision occasionally misreads on individual frames — stray kana prefix, a dakuten flip (こ read as ご), a missing っ. Rather than fixing every individual misread in post-processing, a consensus mechanism requires 4 identical consecutive OCR reads (after normalisation: spaces and punctuation stripped) before the text is considered settled and passed to the LLM. The UI shows "Reading... (2/4)" while building confidence.

A fuzzy dedup variant handles the dakuten ambiguity case: sentences differing by only 1 character count as identical, preventing duplicate LLM calls from the same dialogue box when OCR flickers between こ and ご on consecutive frames.

6. Component — LLM Selection and Call Gating

6.1 When to Call the LLM — The Gate Stack

Knowing when to fire an LLM call was as important as having working OCR. Several independent problems needed solving simultaneously.

Typewriter Animation

Zelda displays dialogue one character at a time. A naive pipeline fires on every frame and generates lessons on incomplete sentences. The stability consensus from section 5.4 handles this.

Gameplay Motion False Triggers

Active play causes constant screen change. Fix: only watch the bottom ~30% of the TV crop (the dialogue zone) rather than the full frame. Link moving in the upper portion of the scene produces no diff in the dialogue zone.

Brightness Gate

Skip OCR entirely if the crop is too bright (open gameplay, no dark dialogue box visible) or too dark (cutscene fade). Constants BRIGHTNESS_GATE_HIGH = 80.0 and BRIGHTNESS_GATE_LOW = 10.0, tuned from real gameplay data in the CSV log.

Hallucination Gate

When no dialogue box is visible, Apple Vision sometimes returns noise — very short strings or isolated characters. A minimum length gate (fewer than 5 Japanese characters) rejects these and resets the stability counter.

The LLM-as-Detector Experiment

The idea of letting the LLM decide when to translate was explored — pass every frame, return ANIMATING / NO_DIALOGUE / DUPLICATE / or a translation. This would eliminate all the pixel diff logic. The problem: 9 seconds per call means 9 seconds just to confirm the screen hasn't changed. Pixel diff runs in microseconds. Abandoned.

CSV Logging for Threshold Tuning

Pixel diff and brightness were logged every frame alongside LLM output. This data revealed a natural three-region pattern in pixel diff values:

Pixel Diff RangeInterpretationAction
Below lower thresholdStatic screen, same dialogue boxSkip
Lower ≤ diff ≤ upperDialogue changed or new box appearedFire LLM
Above upper thresholdFull scene change / gameplay motionSkip

This "Goldilocks zone" means the LLM only fires when change is consistent with dialogue box activity, not general gameplay motion.

Final Gate Order (cheapest to most expensive)

1. Brightness gate         → skip if too bright or too dark
2. Pixel diff check        → only proceed if dialogue zone changed
3. Empty OCR check         → reset stability, skip
4. Minimum length < 5      → reset stability, skip
5. Stability consensus     → skip until 4 identical reads
6. Fuzzy dedup             → skip if ≤1 char different from last
→ Fire LLM

6.2 LLM Selection for Translation

ModelVerdict
LLaVA-Phi3Poor Japanese quality, unnatural English. Eliminated.
qwen2.5:1.5b (~986 MB)Slipped into Chinese mid-sentence on food vocabulary. Eliminated.
qwen2.5:3b (~2 GB)Meaningful improvement but inconsistent on dialect forms.
qwen2.5:7b (~4.7 GB)Final choice. Handles Zelda's archaic NPC speech reliably, captures nuance, produces natural English matching the speaker's register. ~2,500 ms per call.

With Apple Vision handling OCR there's no vision encoder overhead, so the 7b fits comfortably in 16 GB RAM alongside the OS. OLLAMA_FLASH_ATTENTION=1 was set permanently in ~/.zshrc — reduces peak memory via tiled computation. Note: OLLAMA_GPU_LAYERS has no effect on Apple Silicon; it is an NVIDIA/AMD flag.

7. Component — The Learning System

7.1 Design Decisions Upfront

Two parallel pipelines. TRANSLATE runs continuously on its own thread regardless of which tab is visible. LEARN generates lessons and locks the learn panel until acknowledged. Switching between tabs does not pause either pipeline.

Acknowledge-to-proceed. A lesson does not auto-advance. The user presses "Got it" to commit vocabulary to disk and allow the next lesson. This was a deliberate learning design choice — passive streaming of lessons at game speed produces no learning. Making the user actively commit changes the dynamic from "subtitles" to an actual learning interaction.

Queue-one model. If multiple dialogue boxes pass while a lesson is pending, only the most recent is queued. If 5 boxes scroll past, boxes 1–4 are lost and box 5 is queued. This prevents a backlog of unreviewed lessons.

Response options as content. Zelda dialogue includes player-selectable responses. Rather than cropping these out as noise, the lesson includes their translations — choosing a response is part of playing the game.

7.2 The Latency Problem — LLM Doing Everything

The first LEARN implementation asked the LLM to generate the entire lesson in one JSON response: romaji, word-by-word breakdown, grammar notes, kanji analysis, progress notes. Results:

FieldQuality
TranslationConsistently good
RomajiMostly correct — occasional wrong readings that would actively mislead a learner
Word breakdownMissed particles inconsistently
Kanji detectionIncomplete — 7 kanji in a sentence might get 2–3 in output
Grammar notesUnreliable

Generation time: 17,000–35,000 ms per call. Essentially unusable for real-time gameplay. Prompt trimming was tried — removing fields, truncating output, reducing verbosity. None made a meaningful dent. The problem was structural: the LLM was doing systematic linguistic analysis for which it was both slow and unreliable. No amount of prompt engineering fixes a task mismatch.

7.3 The NLP Pivot — Dedicated Libraries

The decisive question: what is the LLM actually good at here? Natural language translation — understanding nuance, dialect, register. Everything else in the lesson is a solved problem with deterministic tools.

fugashi (MeCab morphological analyser)

Splits Japanese text into morphemes with full POS tagging. Knows that こしらえる is a causative volitional verb, は is a topic marker particle, かのう is an archaic dialectal sentence-ender. POS tags are mapped to English labels via _POS_LABELS. Without this, raw MeCab 4-character POS strings leak into the UI — a production bug found when 連体詞 appeared in the breakdown table.

pykakasi

Converts Japanese to romaji. Deterministic. Always correct. Generated per-token from MeCab segmentation, producing clean word boundaries.

jamdict

Dictionary lookups via JMdict and KANJIDIC. Key integration lessons: jamdict-data is a separate pip package from jamdict — installing jamdict alone produces silent "unable to open database file" failures that look like code bugs. Conjugated verb forms have no dictionary entry; fix is to use word.feature.lemma (the MeCab dictionary form) for lookups. Single-character grammar particles are excluded from dictionary lookup via a _SKIP_VOCAB set to avoid wrong homophone matches.

Rule-Based Grammar Detection

Common JLPT N5/N4 patterns are matched against the MeCab POS sequence: te-form progressives (〜ている), polite requests (〜てください), want expressions (〜たい), reason markers (〜から/ので), sentence-ending particles, conditionals. No LLM. No latency. No hallucinations.

Result of the pivot:

Time per learn call
Before NLP pivot17,000–35,000 ms
After NLP pivot~200 ms (NLP) + ~2,500 ms (translation) = ~2,700 ms

7.4 Vocabulary Tracking and Familiarity

Every acknowledged lesson writes its words and kanji to vocab.json. Counting is opt-in — only acknowledged lessons increment times_seen. This reflects genuine engagement: you might see the same NPC line 10 times without ever paying attention to it.

Familiarity uses coloured underlines (not background highlights, which make white text on coloured backgrounds unreadable at large font sizes):

ExposureUnderlineStatus
0 times seen■ BlueNew
1–9 times seen■ YellowLearning
10+ times seen■ GreenFamiliar

Once ≥3 quiz recalls are available, familiarity becomes accuracy-based. A word seen 15 times but consistently missed in quizzes correctly stays in "learning." Exposure alone is not a reliable proxy for retention.

8. Component — The Web UI

8.1 Flask + Polling Architecture

The UI is a single-page HTML template served by Flask. The browser polls /state every 500 ms and updates the DOM in place — no full page refreshes, no WebSockets. The /preview endpoint streams the preprocessed camera frame as an MJPEG response, giving a real-time view of exactly what Apple Vision is receiving.

8.2 TRANSLATE and LEARN Tabs

TRANSLATE mode is always live. Shows Japanese text with familiarity underlines, romaji, and English translation. No lesson, no interruption. For playing at pace.

LEARN mode shows the full lesson: word breakdown table (word / reading / meaning / grammar role / familiarity), grammar pattern card, kanji card with examples, acknowledge bar. Locks until "Got it." Switching tabs does not affect either pipeline. The MJPEG preview is visible only in TRANSLATE mode — seeing live camera output while a frozen lesson was on screen created a visual conflict.

8.3 Metrics and Pipeline Visibility

Per-component timing is tracked and surfaced in the UI:

translate_calls  /  translate_ocr_ms  /  translate_llm_ms
learn_calls      /  learn_ocr_ms      /  learn_nlp_ms  /  learn_llm_ms

These numbers made every architectural change immediately visible: the OCR bar shrank from ~638 ms to ~100 ms when Apple Vision replaced manga-ocr; the learn LLM bar dropped from ~25,000 ms to ~2,500 ms after the NLP pivot.

8.4 The Quiz System

After QUIZ_EVERY acknowledged lessons (default 10), a quiz fires automatically and blocks lesson generation until completed or quit. Cards are built from the N most recent lessons and randomly sampled at n/4 to ensure each quiz is a genuinely different test (the initial version used every word from every lesson, producing 60+ card quizzes). lessons_since_quiz is persisted to disk — without persistence, closing the script would reset the counter and a quiz might never trigger if the user consistently restarted before reaching QUIZ_EVERY.

8.5 Lesson History Sidebar

The last 10 acknowledged lessons are displayed in a sidebar as clickable Japanese text entries. Clicking reopens the full lesson detail. This provides lightweight review of recent dialogue without requiring a quiz.

8.6 UI Polish — Fonts, Colours, Eye Strain

At initial font sizes, kanji on a Retina display at arm's length caused visible eye strain — strokes were too small to read comfortably. Font sizes were increased significantly so the main panel can be read at a glance while playing, not squinted at while sitting at a desk.

The original colour scheme (Catppuccin Mocha: near-black background with vivid saturated accents) caused eye fatigue during extended sessions, particularly when scrolling. Updated to a warm dim palette: dark brown-black background, softer accent colours.

DM Sans (English) and Space Mono (romaji) were replaced with Nunito — a rounded, full-stroke font that reads well at large display sizes. Scroll jitter caused by heavy text reflow was fixed by adding will-change: transform and transform: translateZ(0) to the scroll container, forcing GPU compositing. word-break: break-all was replaced with overflow-wrap: break-word, which handles Japanese text wrapping more naturally.

9. Operations — Startup, Setup, and Persistence

9.1 Startup Script

start_zelda_translator.sh automates the full startup sequence: check Python, check and install missing Python dependencies, check Ollama is installed, check qwen2.5:7b is pulled, check bounds.json exists, launch zelda_translator.py.

An early version used bash associative arrays (declare -A). This failed on macOS with "declare: -A: invalid option" — macOS ships bash 3.2 by default to avoid GPL v3 licensing. Bash associative arrays require bash 4.0+. Fixed by replacing with simple paired-list logic. The script is macOS-only — pyobjc-framework-Vision does not exist on Windows.

9.2 Model Unload on Exit

Ollama keeps models loaded in unified memory after use. On a 16 GB M1 Pro, failing to unload the 7b model after exit wastes ~4.7 GB. A finally block in the entry point explicitly unloads on Ctrl+C:

def unload_model():
    requests.post(OLLAMA_URL,
                  json={"model": TRANSLATION_MODEL, "keep_alive": 0})

10. Lessons Learned

Use the right tool for each layer

The single biggest improvement came from stopping the attempt to use one model for everything. Apple Vision for OCR — built-in, zero RAM, Neural Engine, ~100 ms. Deterministic NLP libraries for linguistic analysis — fast and more accurate than a 7b model at segmentation, dictionary lookup, and POS tagging. LLM for translation only — one task it does reliably. Each layer became more accurate and faster when doing what it was designed for.

Hardware constraints drive architecture

16 GB unified memory sounds like plenty until a vision LLM allocates 14 GB for weights and a KV cache. Every major architecture decision — dropping the vision LLM, splitting OCR from translation, choosing text-only models — was driven by hitting RAM limits on real hardware. The architecture is not an abstract design; it is the shape that fits inside the machine.

Physical setup is part of the system

Moving the camera closer to the TV was more effective than any amount of image preprocessing at eliminating OCR errors. Better source material beats better compensation downstream.

Log everything while tuning

CSV logging of brightness, pixel diff, OCR output, and timing at every frame was essential for finding workable threshold values. The debug habit of saving the preprocessed frame to disk on every iteration saved hours of guessing about why OCR was behaving unexpectedly.

Gate early, gate hard

The multi-gate validation pipeline eliminates nearly all unnecessary LLM calls at microsecond cost. The ordering matters: reject the cheapest failures first. Each gate prevents the next from running. The LLM call is the last resort, not the first.

User pacing is a feature

The acknowledge-to-proceed flow forces the system to wait for the user. This felt like a constraint during design but became one of the most important decisions — passive streaming of lessons at game speed produces no learning. Making the user actively commit before moving on changes the dynamic from "translator with subtitles" to an actual learning tool.
